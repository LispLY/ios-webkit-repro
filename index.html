iOS 26.5 WKWebView crash reproduction

Reproduces Apple WebKit bug 317910 ("WebContent process termination due to MESSAGE_CHECK in WebProcessProxy::didPostMessage()"). Affects iOS/iPadOS 26.5 and 26.5.1 in WKWebView browsers that inject content scripts (Edge, Chrome). iOS 26.4.x and earlier are not affected; Safari is not affected; Apple fixed it in a later WebKit.

Open each case in Edge or Chrome on an iOS 26.5 device. Each page tries the trigger automatically on load; if the popup was blocked (no user gesture), tap the button on the page. Reproduced = the tab crashes to "Can't open this page" (sad tab). Not reproduced = the page keeps logging and stays alive.

Case A — inside an <iframe> →

The trigger runs in a subframe: an iframe whose script calls window.open('about:blank') then closes it. This is the minimal known trigger.

Case B — top-level, no iframe →

Mirrors the customer's actual code: a top-level page that on load does a 1×1 window.open popup-blocker probe, then a second "real document" popup — no iframe. Tests whether the crash needs an iframe at all.

Why two cases: the confirmed WebKit bug reproduces with an iframe (Case A). The customer's pasted code is a top-level probe with no iframe (Case B). Comparing the two on a real device shows whether the iframe is required, or whether the customer's top-level sequence triggers the same crash — i.e. whether this repro represents the customer's actual scenario.
