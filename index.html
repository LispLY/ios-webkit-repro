iOS 26.5 WKWebView crash reproduction

Reproduces Apple WebKit bug 317910 ("WebContent process termination due to MESSAGE_CHECK in WebProcessProxy::didPostMessage()"). It affects iOS/iPadOS 26.5 and 26.5.1 in any WKWebView-based browser that injects content scripts (e.g. Edge and Chrome for iOS). iOS 26.4.x and earlier are not affected; Safari is not affected; Apple fixed it in a later WebKit.

How to use: open this page in Edge or Chrome for iOS on an iOS 26.5 device. The subframe below tries the trigger automatically on load; if the popup was blocked (no user gesture), tap the button inside the frame. Reproduced = the tab crashes to "Can't open this page" (sad tab). Not reproduced = the frame keeps logging and the tab stays alive.

Mechanism: the frame calls window.open('about:blank') and closes it immediately. On iOS 26.5 this races with WebKit's page teardown; a benign content-script postMessage in flight fails an over-strict UI-process check and WebKit terminates the render process. The popup content is irrelevant — no PDF, redirect, or SSO is needed.
